# Интеграция Maxma, Tilda и 1С для ООО "ЛАПАЛОМА"

## Цель проекта

Построить решение, в котором:

- `Tilda` используется как текущая витрина и точка оформления заказа.
- `Maxma` отвечает за покупателей, карты лояльности, промокоды, подарочные карты и покупки.
- `1С 8.3` используется как мастер-источник по товарам, параметрам и части операционных данных.
- `Next.js` используется как frontend-слой для развития проекта и постепенного перехода с ограничений `Tilda` на более гибкую архитектуру.
- Развертывание frontend выполняется через `GitHub` + `Vercel`.

## Краткий вывод по реализуемости

Да, такой сценарий реализуем, но для стабильной работы нужен не прямой обмен только между `Tilda` и `Maxma`, а промежуточный backend-слой интеграции.

Рекомендуемая схема:

`Tilda / Next.js frontend -> Backend API -> Maxma API / 1С API / Database`

Это нужно для того, чтобы:

- централизованно проверять номер телефона, карту лояльности, промокод и подарочную карту;
- корректно пересчитывать корзину;
- логировать все запросы и ответы;
- повторно отправлять неуспешные операции;
- синхронизировать каталог из `1С`;
- избежать зависимости от ограничений клиентского кода в `Tilda`.

## Целевая архитектура

### Frontend

`Next.js`:

- каталог;
- карточка товара;
- корзина;
- checkout;
- личный кабинет;
- формы авторизации и регистрации;
- интеграция с backend по API.

На первом этапе возможно оставить `Tilda` как маркетинговые страницы и перевести checkout/API-логику на внешний backend.

### Backend

Backend-сервис отвечает за:

- создание и обновление покупателей в `Maxma`;
- поиск клиента по телефону;
- получение карты лояльности;
- проверку и применение промокода;
- проверку и применение подарочной карты;
- пересчет заказа;
- отправку заказа в `Maxma`;
- прием и обработку данных из `1С`;
- синхронизацию товарного каталога;
- журналирование ошибок и интеграционных событий.

Технологически можно использовать:

- `Node.js` + `NestJS` или `Express`;
- очередь задач для обмена с внешними системами;
- `PostgreSQL` для служебных таблиц интеграции;
- `Redis` при необходимости кэширования и очередей.

### Database

База данных нужна не как мастер-источник товаров вместо `1С`, а как служебный слой:

- таблица соответствий товаров `1С <-> Tilda/Next.js`;
- таблица логов запросов в `Maxma` и `1С`;
- таблица очередей синхронизации;
- таблица ошибок импорта;
- кэш промо-данных и промежуточных расчетов, если это потребуется.

## Бизнес-сценарии

### Сценарий 1. Новый клиент

#### Что должно происходить

На стороне `Tilda` или `Next.js` клиент вводит:

- ФИО;
- телефон;
- email;
- номер карты лояльности, если уже есть;
- адрес.

После отправки формы backend:

1. валидирует обязательные поля;
2. ищет клиента по телефону в `Maxma`;
3. если клиента нет, создает нового покупателя;
4. получает из `Maxma` номер карты лояльности;
5. возвращает данные клиента обратно во frontend.

#### Результат

- в `Maxma` создается карточка покупателя;
- клиенту автоматически присваивается карта лояльности;
- во frontend можно сразу показать номер карты или статус участника программы.

### Сценарий 2. Покупки (заказы)

#### Что должно происходить

Клиент оформляет заказ, backend:

1. собирает состав корзины;
2. прикладывает скидки, промокоды, подарочные карты;
3. фиксирует итоговую сумму;
4. отправляет заказ в `Maxma`;
5. получает идентификатор покупки и статус.

#### Результат

- заказ автоматически появляется в `Maxma` как покупка "в работе";
- в заказе сохраняются все примененные скидки и основания пересчета;
- backend хранит технический лог обмена.

### Сценарий 3. Карта лояльности

#### Что должно происходить

Во время checkout клиент вводит телефон.

Backend:

1. отправляет запрос в `Maxma`;
2. получает данные клиента и карты лояльности;
3. определяет размер скидки или бонусные условия;
4. возвращает обновленные данные корзины.

#### Результат

- номер карты подтягивается автоматически;
- сумма корзины пересчитывается до оплаты;
- в заказе сохраняется факт применения карты.

### Сценарий 4. Промокод

#### Что должно происходить

Клиент вводит промокод и нажимает кнопку активации.

Backend:

1. проверяет формат промокода;
2. обращается в `Maxma`;
3. проверяет срок действия, статус и ограничения;
4. рассчитывает скидку;
5. возвращает обновленную корзину.

#### Результат

- промокод валидируется в реальном времени;
- скидка отображается сразу;
- в `Maxma` и в заказе сохраняется корректный пересчет.

### Сценарий 5. Подарочная карта

#### Что должно происходить

Клиент вводит ФИО, телефон и номер подарочной карты.

Backend:

1. проверяет карту в `Maxma`;
2. получает остаток и условия применения;
3. применяет допустимую сумму списания;
4. пересчитывает итог заказа;
5. передает результат во frontend и затем в заказ.

#### Результат

- подарочная карта активируется по кнопке;
- сумма корзины уменьшается корректно;
- в `Maxma` покупка сохраняется с учетом списания.

## Ключевое требование по расчетам

Во время оформления заказа должен быть оперативный обмен между frontend и backend, а backend уже работает с `Maxma`.

Критично реализовать единый модуль расчета заказа, который учитывает:

- базовую стоимость товаров;
- скидку по карте лояльности;
- промокод;
- подарочную карту;
- правила совместимости скидок;
- итоговую сумму к оплате;
- состав передаваемых данных в `Maxma`.

На текущий момент согласованы такие правила совместимости скидок:

- можно совмещать карту лояльности и сертификат;
- можно совмещать промокод и сертификат;
- карта лояльности и промокод как скидки не суммируются;
- комбинация карта лояльности + промокод + сертификат не разрешена;
- порядок применения при допустимой комбинации: сначала скидка, потом сертификат.

Именно backend должен быть "истиной" для итогового расчета, а не только интерфейс корзины.

Если телефон клиента не найден в `Maxma`, во время checkout нужно показать предложение зарегистрироваться в программе лояльности.

Текущий согласованный сценарий регистрации:

- предложение о регистрации показывается в checkout;
- переход в регистрацию должен вести через `Telegram`-бота;
- у клиента должна быть возможность отказаться от регистрации через небольшое окно отказа.

Если `Maxma` недоступна во время оформления заказа, заказ должен продолжаться без применения скидок.

В checkout нужно явно показывать клиенту сумму "до" и итоговую сумму "после всех скидок".

Под фразой "данные уходят в заказ" имеется в виду, что в заказ, который backend отправляет в `Maxma`, должны попасть не только товары и итоговая сумма, но и основания пересчета.

Пример состава данных, которые должны быть зафиксированы в заказе:

- клиент и телефон;
- список товаров;
- базовая сумма до скидок;
- итоговая сумма после скидок;
- номер карты лояльности, если использована;
- промокод, если использован;
- номер сертификата, если использован;
- сумма списания по сертификату;
- сумма скидки по карте лояльности или промокоду.

## Интеграция 1С -> Tilda / Next.js

### Возможно ли это

Да, такой сценарий возможен.

Лучше строить его не как прямую зависимость `1С -> Tilda`, а как:

`1С -> Backend -> Database -> Next.js/Tilda`

### Какие данные можно передавать из 1С

Из карточки товара можно синхронизировать:

- описание;
- заголовок;
- бренд;
- артикул;
- фото;
- вес;
- габариты;
- цвет;
- размер.

### Логика синхронизации

При обмене:

- если товара с таким артикулом нет, создается новая карточка;
- если товар уже есть, выполняется обновление и дополнение полей;
- артикул используется как основной внешний ключ сопоставления;
- изображения и параметры проходят отдельную проверку качества и полноты.

Дополнительно зафиксировано бизнес-правило:

- основная выгрузка товаров идет из `1С`;
- если карточка была скорректирована вручную в `Tilda`, эти изменения должны сохраняться на сайте `Tilda`;
- при этом, если позже данные в `1С` снова обновлены, на `Tilda` тоже должно прийти обновление;
- значит, желаемое правило синхронизации: приоритет по времени последнего изменения (`last write wins`), если оно технически подтверждается по источникам данных;
- если товара больше нет в `1С`, он должен скрываться на стороне `Tilda`.

### Ограничения и риски

У вас уже есть важные вводные:

- `1С 8.3`;
- около `3000` товаров;
- фото в `1С` плохого качества и не везде заполнены;
- часть фото есть на старом сайте на `Bitrix`;
- регламент ведения карточек пока не сформирован.

Из этого следует:

1. перед автоматическим импортом нужно подготовить единый стандарт карточки товара;
2. нужно определить приоритет источников фото:
   - сначала новый фотобанк;
   - затем фото со старого сайта;
   - затем фото из `1С`, если они проходят по качеству;
3. нужен этап нормализации данных по цветам, размерам, весу и габаритам;
4. нужен справочник соответствий для брендов, свойств и категорий;
5. нужен технический отчет по товарам, которые не удалось загрузить автоматически.

## Предлагаемая структура проекта

```text
database/
frontend/
backend/
IMPLEMENTATION_PLAN.md
```

### Назначение папок

`frontend`:

- приложение на `Next.js`;
- интерфейсы каталога, корзины, checkout и личного кабинета;
- интеграция с backend API.

`backend`:

- API для frontend;
- интеграции с `Maxma`;
- интеграции с `1С`;
- бизнес-логика расчета заказа;
- логирование и очереди.

`database`:

- схема БД;
- миграции;
- справочники соответствий;
- служебные данные интеграции.

## Развертывание

### Frontend

- код хранится в `GitHub`;
- frontend разворачивается на `Vercel`;
- для каждой ветки можно использовать preview deployment;
- production-сборка публикуется из основной ветки.

### Backend

Backend лучше размещать отдельно от `Vercel frontend`, например:

- `Railway`;
- `Render`;
- `Yandex Cloud`;
- `Selectel`;
- `Docker` на VPS.

Это удобнее для:

- защищенных интеграций;
- фоновых задач;
- cron-процессов;
- очередей;
- стабильной работы webhook/API.
