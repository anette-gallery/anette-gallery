# API-контракты интеграции Maxma, Tilda и 1С

## Назначение

Документ фиксирует стартовые API-контракты между `frontend`, `backend`, `Maxma`, `Tilda` и `1С`.
Все внешние интеграции идут через backend, чтобы расчет корзины и бизнес-правила жили в одном месте.

Базовый префикс backend API:

```text
/api/v1
```

## Общие принципы

- формат обмена: `JSON`;
- идентификатор клиента: телефон в нормализованном формате;
- идентификатор товара: артикул или `sku`;
- итоговый расчет заказа выполняется на backend;
- `Tilda` и `Next.js frontend` не рассчитывают скидки как источник истины;
- все операции интеграции должны логироваться.

## 1. Клиенты и Maxma

### POST `/api/v1/customers/sync`

Создает клиента в `Maxma` или обновляет существующего по номеру телефона.

Пример запроса:

```json
{
  "fullName": "Анна Волкова",
  "phone": "+79990000000",
  "email": "anna@example.com",
  "loyaltyCardNumber": "",
  "address": "Москва, ул. Пример, 10"
}
```

Пример ответа:

```json
{
  "status": "stub",
  "target": "maxma",
  "action": "sync-customer",
  "payload": {
    "fullName": "Анна Волкова",
    "phone": "+79990000000",
    "email": "anna@example.com",
    "loyaltyCardNumber": "",
    "address": "Москва, ул. Пример, 10"
  }
}
```

## 2. Checkout и лояльность

Единый объект корзины:

```json
{
  "phone": "+79990000000",
  "promoCode": "SPRING10",
  "giftCardNumber": "GIFT-001",
  "items": [
    {
      "sku": "SKU-001",
      "quantity": 1,
      "price": 5400
    }
  ]
}
```

Текущие согласованные правила совместимости скидок:

- карта лояльности + сертификат: разрешено;
- промокод + сертификат: разрешено;
- карта лояльности + промокод: не суммируются;
- карта лояльности + промокод + сертификат: не разрешено;
- порядок применения при допустимой комбинации: сначала скидка, потом сертификат.

### POST `/api/v1/checkout/loyalty/apply`

Получает карту лояльности по телефону и пересчитывает корзину.

Если телефон не найден в `Maxma`, backend должен вернуть признак, что клиенту нужно предложить регистрацию в программе лояльности.

Текущий согласованный канал регистрации:

- предложение о регистрации должно вести в `Telegram`-бот;
- клиент должен иметь возможность отказаться от регистрации через небольшое окно отказа.

### POST `/api/v1/checkout/promo-code/validate`

Проверяет промокод в `Maxma` и возвращает обновленный расчет.

### POST `/api/v1/checkout/gift-card/validate`

Проверяет подарочную карту и сумму доступного списания.

### POST `/api/v1/checkout/calculate`

Выполняет финальный серверный расчет корзины с учетом всех скидок.

Если `Maxma` недоступна, оформление заказа должно оставаться возможным, но без применения скидок.

Во frontend нужно показывать:

- базовую сумму до скидок;
- итоговую сумму после всех скидок;
- сумму списания по сертификату и сумму скидки по каждому примененному механизму.

Базовый ответ на этом этапе:

```json
{
  "status": "stub",
  "action": "calculate-checkout",
  "payload": {
    "phone": "+79990000000",
    "promoCode": "SPRING10",
    "giftCardNumber": "GIFT-001",
    "items": [
      {
        "sku": "SKU-001",
        "quantity": 1,
        "price": 5400
      }
    ]
  }
}
```

## 3. Заказы

### POST `/api/v1/orders`

Создает заказ на backend и отправляет его в `Maxma` как покупку `в работе`.

Под фразой "данные уходят в заказ" имеется в виду, что в заказ должны передаваться не только товары и итоговая сумма, но и данные о примененных скидках.

Пример запроса:

```json
{
  "customer": {
    "fullName": "Анна Волкова",
    "phone": "+79990000000",
    "email": "anna@example.com"
  },
  "items": [
    {
      "sku": "SKU-001",
      "quantity": 1,
      "unitPrice": 5400
    }
  ],
  "totalAmount": 4900,
  "promoCode": "SPRING10",
  "giftCardNumber": "GIFT-001",
  "loyaltyCardNumber": "1234567890"
}
```

В рабочем варианте заказ должен фиксировать:

- данные клиента;
- товары и количество;
- базовую сумму до скидок;
- итоговую сумму после скидок;
- номер карты лояльности, если использована;
- промокод, если использован;
- номер сертификата, если использован;
- размер скидки или списания по каждому механизму.

## 4. Каталог и 1С

### POST `/api/v1/catalog/sync`

Синхронизирует одну карточку товара из `1С`.

Пример запроса:

```json
{
  "sku": "SKU-001",
  "title": "Платье Lapaloma",
  "description": "Летнее платье",
  "brand": "Lapaloma",
  "color": "Белый",
  "size": "S",
  "weight": 0.3,
  "dimensions": "30x20x5",
  "images": [
    "https://cdn.example.com/products/sku-001-1.jpg"
  ]
}
```

### POST `/api/v1/catalog/sync/batch`

Синхронизирует пакет товаров из `1С`.

Пример запроса:

```json
{
  "items": [
    {
      "sku": "SKU-001",
      "title": "Платье Lapaloma"
    },
    {
      "sku": "SKU-002",
      "title": "Жакет Lapaloma"
    }
  ]
}
```

## 5. Tilda -> Backend

Рекомендуемый подход:

- формы и checkout в `Tilda` отправляют запросы в backend;
- backend валидирует клиента, лояльность, промокод и сертификат;
- backend возвращает пересчитанную корзину;
- после подтверждения backend создает заказ и отправляет его в `Maxma`.
- если клиента по телефону нет в `Maxma`, frontend на `Tilda` должен показать предложение зарегистрироваться в программе лояльности.
- регистрация должна предлагаться через `Telegram`-бот.

## 6. Backend -> Maxma

На этапе реализации нужно уточнить реальную документацию `Maxma`, но на уровне контракта backend должен поддержать:

- поиск клиента по телефону;
- создание клиента;
- получение карты лояльности;
- валидацию промокода;
- валидацию подарочной карты;
- создание покупки;
- обновление статуса заказа при необходимости.

## 7. Backend -> 1С

На этапе реализации нужно зафиксировать способ интеграции с `1С 8.3`:

- HTTP API;
- web-сервисы `1С`;
- обмен через регламентные задания;
- промежуточная выгрузка файлов, если API отсутствует.

Минимальные поля для обмена каталогом:

- `sku`;
- `title`;
- `description`;
- `brand`;
- `images`;
- `weight`;
- `dimensions`;
- `color`;
- `size`.

Дополнительно согласовано бизнес-правило по каталогу:

- основная выгрузка товаров идет из `1С`;
- все поля в `Tilda` могут редактироваться вручную;
- `Tilda` должна иметь приоритет для быстрых правок;
- если позднее данные в `1С` снова обновлены, на `Tilda` тоже должно прийти обновление;
- желаемое правило синхронизации: приоритет по времени последнего изменения (`last write wins`);
- если товара больше нет в `1С`, он должен скрываться в `Tilda`.
